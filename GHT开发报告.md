## 一、项目概述

### 1.1 项目愿景
为 GHT（Great Himalaya Trail）大喜马拉雅高线徒步者打造一个**实时轨迹追踪 + 进度可视化 + 日志记录 + 捐助支持**的一站式平台。徒步者的朋友、家人和支持者可以通过网页实时看到徒步进度，了解当前位置和关键地理信息。

### 1.2 核心功能清单

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| F1 | 尼泊尔区域版图 | P0 | 7省区划 + 当前所在区域高亮 |
| F2 | GHT预设轨迹（计划路线） | P0 | 第一次导入的完整GPX作为基准路线 |
| F3 | 实际轨迹上传（进度记录） | P0 | 每天/隔几天上传GPS轨迹，与预设路线对比 |
| F4 | 进度可视化 - 徒步小人 | P0 | 地图上显示当前行进位置，用动画小人表示 |
| F5 | 进度可视化 - 颜色分段 | P0 | 已完成段绿色，当前段蓝色，未走段灰色 |
| F6 | 关键点标注 | P0 | 垭口(12个)、住宿点(7个)、观景点(5个) |
| F7 | 累计爬升/下降 | P1 | 基于实际GPS数据动态计算 |
| F8 | 徒步日志 | P0 | 带日期+照片的文字日志系统 |
| F9 | 捐助通道 | P1 | 微信/支付宝收款码展示 |
| F10 | 数据持久化 | P1 | 日志和轨迹存储在云端 |

## 二、微信小程序 vs 网站 — 对比分析

| 维度 | 微信小程序 | Web 网站 (PWA) |
|------|-----------|----------------|
| **地图渲染** | 需使用腾讯地图 SDK，功能受限 | Leaflet/MapboxGL，功能强大、生态丰富 |
| **GPX解析** | 需自行实现，文件系统限制 | FileReader API，轻松读取大文件 |
| **分享传播** | ✅ 微信生态天然优势，朋友圈/群聊一键分享 | 需主动保存链接/二维码 |
| **捐助流程** | ✅ 微信支付原生接入，体验极佳 | 需要跳转微信扫码，多一步 |
| **开发成本** | 需微信审核+域名备案+HTTPS | 部署即上线，无审核 |
| **离线能力** | ✅ 小程序原生支持 | 需 PWA Service Worker 配置 |
| **国际访问** | ❌ 仅微信内打开，国外用户无法访问 | ✅ 全球可访问，不依赖任何平台 |
| **迭代速度** | 审核周期 1-7天 | 即时部署 |
| **日志功能** | 需自建后端 | 可使用 localStorage / 后端 API |
| **地图自定义** | 受限（仅限腾讯地图图层） | 完全自由（可选任何地图源） |

### 🏆 推荐方案：**Web 网站 (PWA) + 微信内嵌网页**

**理由**：
1. **国际可达**：尼泊尔徒步期间可能有国外朋友关注，Web不受平台限制
2. **地图能力**：Leaflet + OpenStreetMap 免费且强大，支持 GPX 图层叠加
3. **快速迭代**：不需要微信审核，11月出发前可以持续快速优化
4. **捐助折中**：通过网页展示微信/支付宝收款码，用户扫码捐助（多一步但可行）
5. **PWA 加持**：可添加到手机主屏幕，离线缓存地图，体验接近原生App

如果后续需要更好的捐助体验，可以将捐助页面单独做成微信小程序，主体保留为网站。

## 四、技术方案

### 4.1 技术栈

```
前端：HTML5 + CSS3 + JavaScript (Vanilla or Vue3)
地图：Leaflet.js + OpenStreetMap / CartoDB 暗色主题
GPX解析：@mapbox/togeojson (开源库)
数据存储：localStorage (离线) + 可选云数据库
部署：CloudStudio / Vercel / GitHub Pages
PWA：Service Worker + Manifest.json (离线支持)
```

### 4.2 架构设计

```
┌──────────────────────────────────────────────┐
│                   浏览器端                      │
│  ┌────────┐ ┌──────────┐ ┌───────────────┐   │
│  │ Map    │ │ Sidebar  │ │ File Upload   │   │
│  │(地图)  │ │(侧边栏)  │ │(GPX上传)      │   │
│  └────┬───┘ └────┬─────┘ └───────┬───────┘   │
│       │          │               │           │
│  ┌────┴──────────┴───────────────┴───────┐   │
│  │         State Manager (App State)      │   │
│  └───────────────────┬───────────────────┘   │
│                      │                        │
│  ┌───────────────────┴───────────────────┐   │
│  │    localStorage (日志+轨迹+进度)       │   │
│  └───────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

### 4.3 数据流

```
【第一次上传】→ 设为预设轨迹
GPX文件 → FileReader → GPX解析器
  → [lat, lng, ele, time]数组
  → 存储为 presetTrack（预设轨迹）
  → 显示：灰色虚线（底层）

【后续多次上传】→ 实际轨迹段
GPX文件 → FileReader → GPX解析器
  → [lat, lng, ele, time]数组
  → 自动匹配到预设轨迹的位置（最近点算法）
  → 存储为 actualTracks[]（实际轨迹段数组）
  → 显示：绿色实线（已走段）+ 蓝色实线（当前段）

【进度计算】
最后一次上传的轨迹终点 → 在预设轨迹上找对应位置
  → 完成百分比 = (预设轨迹上已走距离 / 预设轨迹总长度) × 100%
  → 徒步小人位置 = 最后一次上传轨迹的终点

【日志关联】
每次上传可附带日志 → 存储为 logEntries[]
  → 点击轨迹段 → 显示对应日志
```

### 4.4 数据模型设计

#### 前端存储结构（localStorage）


## 五、技术难点及解决方案

### 难点 1：GPX轨迹解析与统计计算 ⭐⭐⭐

**问题**：GPX文件包含大量GPS点，需要解析后计算总距离（WGS84大地线）、爬升/下降。

**方案**：
- 使用 `@mapbox/togeojson` 将 GPX 转 GeoJSON
- 使用 **Haversine公式** 计算两GPS点间距
- 爬升/下降算法：遍历GPS点，当海拔差 > 阈值（如10m）时累加
- 阈值过滤避免GPS漂移产生虚假爬升

### 难点 2：尼泊尔偏远地区离线使用 ⭐⭐⭐⭐

**问题**：尼泊尔GHT经过大量无网络区域，页面必须能离线工作。

**方案**：
- **PWA Service Worker**：预缓存地图切片（徒步区域的 8-13 zoom level）
- **LocalStorage**：所有轨迹数据先在本地存储
- **后台同步**：有网络时自动上传到云端
- 地图切片预缓存可通过 Leaflet 的 `L.TileLayer.Offline` 插件实现

### 难点 3：预设轨迹 vs 多次上传的实际轨迹对比 ⭐⭐⭐

**问题**：用户会在徒步过程中多次上传GPX文件（每天或隔几天），需要将多次上传的轨迹与第一次导入的预设轨迹进行叠加对比，并显示进度。

**方案**：
- **数据结构设计**：
  - `presetTrack`：第一次上传的完整GPX，作为基准路线
  - `actualTracks[]`：数组，存储多次上传的实际轨迹段
  - 每次上传时，自动找到该段在预设轨迹上的对应位置（通过最近点匹配）
- **可视化策略**：
  - 预设轨迹：灰色虚线（底层）
  - 实际轨迹：绿色实线（已走段）+ 蓝色实线（当前段）
  - 未走段：半透明灰色
- **进度计算**：
  - 找到最后一次上传轨迹的终点
  - 在预设轨迹上找到对应位置
  - 计算完成百分比 = (预设轨迹上已走距离 / 预设轨迹总长度) × 100%

### 难点 4：徒步小人动画实现 ⭐⭐⭐⭐

**问题**：需要在地图上显示一个"徒步小人"图标，表示当前行进位置，并且可以有简单的动画效果。

**方案**：
- **图标设计**：
  - 使用 emoji 🚶 或自定义 SVG 图标（小人背着包）
  - 图标始终显示在"最后一次上传轨迹的终点"位置
- **动画效果**（可选）：
  - CSS animation：让小人有"行走"效果（左右摆动）
  - 使用 Leaflet 的 `L.marker` + 自定义 icon
  - 可以加一个脉冲光环（CSS pulse animation）表示"当前位置"
- **交互**：
  - 点击小人，显示"当前状态"（今天走了多少公里、海拔、日志摘要）
  - 小人随每次上传自动移动到新位置

### 难点 5：大量GPS点性能优化 ⭐⭐

**问题**：150天徒步，每天可能产生数千个GPS点，加上多次上传，数据量会很大。

**方案**：
- 使用 Douglas-Peucker 算法简化轨迹
- 地图缩放级别自适应：zoom<10显示简化线，zoom≥10显示详细线
- 多次上传的轨迹段分别存储，每次只加载当前段+前后段的详细数据
- 使用 Leaflet 的 `L.layerGroup` 分层管理

## 七、UX 建议：信息展示方式

### 问题：关键点信息（垭口、住宿点、观景点、公里数、累计爬升/下降）应该在地图上显示还是单独显示？

### 👉 建议：**双层展示策略**

#### 第一层：地图上（空间感知）
- **徒步小人** 🚶：显示当前位置，带脉冲光环动画
- **垭口**：⛰️ 图标 + 海拔标签 → 点击弹出详细信息
- **住宿点**：🏕️ 图标 + 名称
- **观景点**：📸 图标
- **轨迹颜色分段**：
  - 绿色实线：已完成段
  - 蓝色实线：当前段（最后一次上传）
  - 灰色虚线：未走段（预设轨迹）

#### 第二层：侧边栏（数据阅读）
- **统计卡片**：总距离 | 已完成距离 | 累计爬升 | 累计下降 | 当前海拔 | 剩余天数
- **路段列表**：点击切换视图，地图自动定位
- **详细表格**：展开具体垭口/住宿点的海拔、距离下一站等信息
- **上传记录**：显示所有已上传的轨迹段，点击可回顾

#### 徒步小人动画设计（新增）

**视觉设计**：
- 图标：🚶 或自定义 SVG（小人背包）
- 位置：最后一次上传轨迹的终点
- 动画效果：
  - CSS pulse：脉冲光环（表示"当前位置"）
  - 可选：轻微左右摆动（表示"行走中"）

**交互设计**：
- 点击小人 → 弹出信息窗：
  - 当前海拔
  - 今天已走距离
  - 最近一条日志摘要
  - "查看详情"按钮 → 跳转到对应日志
- 自动跟随：每次上传新轨迹后，小人自动移动到新位置

**技术实现**：
- Leaflet `L.marker` + 自定义 icon（divIcon）
- CSS `@keyframes pulse` 实现脉冲动画
- 位置更新：每次 `handleGPXUpload` 成功后，重新计算最后一次上传的终点

#### 为什么这样设计？

| 信息类型 | 地图上 | 侧边栏 | 原因 |
|---------|--------|--------|------|
| 徒步小人（当前位置）| ✅ 动画图标 | ✅ 统计卡片 | 空间位置+状态信息互补 |
| 垭口位置 | ✅ 标记 | ✅ 列表 | 空间位置+详细数据互补 |
| 住宿点 | ✅ 标记 | ✅ 列表 | 需要看相对位置规划行程 |
| 观景点 | ✅ 标记 | ⚠️ 简要 | 位置比数据更重要 |
| 轨迹颜色分段 | ✅ 可视化 | ✅ 进度条 | 直观展示进度 |
| 公里数 | ❌ | ✅ 卡片 | 纯数字不适合地图显示 |
| 累计爬升/下降 | ❌ | ✅ 卡片 | 动态统计数据 |
| 进度百分比 | ❌ | ✅ 进度条 | 全局指标 |

**核心原则**：地图回答「在哪里」，侧边栏回答「多少/多远」。两者各司其职，不互相污染。

## 八、成本与积分估算

### 8.1 积分（Token）消耗估算

| 阶段 | 任务 | 预估积分 |
